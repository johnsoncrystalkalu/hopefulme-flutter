1. PROJECT OVERVIEW
===================

HopefulMe is a feature-based Flutter mobile application that connects to the HopefulMe backend API and web platform. The app combines authentication, social feed features, messaging, profiles, groups, notifications, library content, inspirational content, search, and deep-link driven navigation.

The project is structured for modular growth:

- app-level setup lives in lib/app
- reusable platform and shared code lives in lib/core
- product modules live in lib/features
- static assets live in assets
- platform runners live in android, ios, web, windows, linux, and macos

At a high level, the app works like this:

1. Flutter starts in lib/main.dart.
2. HopefulMeApp bootstraps global services and repositories.
3. The app restores theme and auth session.
4. If the user is authenticated, the app shows the main HomeScreen shell.
5. If the user is not authenticated, the app shows the welcome/auth flow.
6. While running, the app handles:
   - version checks
   - OneSignal push notifications
   - app deep links
   - user presence pinging
   - cached API-backed content

5. APPLICATION STARTUP FLOW
===========================

5.1 Entry point

File:

Responsibilities:
- ensures Flutter bindings are initialized

5.2 Root app

File:
- lib/app/app.dart

HopefulMeApp is the real application bootstrapper. It is a StatefulWidget because it owns app-wide services, listeners, lifecycle hooks, timers, and repository instances.

Main responsibilities of HopefulMeApp:

- create and hold the Navigator key
- restore theme choice
- load environment configuration
- create TokenStorage and PageCache
- create ApiClient
- create AuthController
- create feature repositories
- initialize deep links
- initialize OneSignal
- run app version checks
- keep online presence updated
- decide whether to show loading, auth, or home UI

Important global objects created here:

- ThemeController
- AuthController
- FeedRepository
- ContentRepository
- ProfileRepository
- NotificationRepository
- MessageRepository
- GroupRepository
- UpdateRepository
- SearchRepository
- LibraryRepository

Why this matters:
- the app is using manual dependency wiring instead of Provider, Riverpod, Bloc, or GetX
- dependencies are passed down through constructors
- this makes dependency flow explicit, but it also means app.dart and large screens can become heavy if not kept disciplined

8. AUTHENTICATION AND SESSION MANAGEMENT
========================================

Files:
- lib/features/auth/data/auth_repository.dart
- lib/features/auth/presentation/controllers/auth_controller.dart
- lib/features/auth/models/
- lib/features/auth/presentation/screens/
- lib/core/storage/token_storage.dart

8.1 AuthRepository

AuthRepository wraps the lower-level ApiClient and exposes auth-specific operations:

- login
- register
- forgot password request
- reset password
- fetch current user
- check username availability
- fetch registration roles
- ping presence
- create web session bridge links
- logout
- register OneSignal player ID
- local session helpers

8.2 AuthController

AuthController is a ChangeNotifier that manages auth state for the UI.

State it owns:
- isLoading
- isBootstrapping
- isAuthenticated
- errorMessage
- currentUser

Important flow:

1. restoreSession() checks if a token exists.
2. If a cached user exists, it uses it immediately.
3. It then refreshes the session in the background.
4. If unauthorized, it clears local session state.
5. UI rebuilds based on notifier changes.

Why this is useful:
- boot is faster because cached user data can render before a fresh auth/me call returns
- the app remains resilient when the network is temporarily unavailable

8.3 Local storage used for auth

The project stores:
- auth token
- cached user object
- theme mode
- page cache entries

This makes the app partly offline-tolerant for already visited pages.


9. NETWORK LAYER
================

Files:
- lib/core/network/api_client.dart
- lib/core/network/api_exception.dart
- lib/core/network/image_url_resolver.dart

9.1 ApiClient responsibilities

ApiClient is the shared HTTP gateway for the application.

Supported request types:
- GET
- POST
- PATCH
- PUT
- DELETE
- multipart POST
- multipart PUT via POST + _method=PUT

Key behaviors:
- injects Authorization header when token exists
- sets Accept and Content-Type headers
- enforces request timeout
- normalizes base URL and request paths
- decodes JSON safely
- handles common HTTP errors with user-friendly messages
- throws ApiException for UI-friendly error handling

9.2 Error handling approach

The network layer does a good amount of user-facing safety work:

- timeout errors are translated into readable messages
- transport errors are translated into readable messages
- non-JSON error responses are handled
- validation errors are extracted from the API error payload
- common status codes have friendly fallback messages

This means repositories and screens can stay cleaner because the low-level formatting of failures is already handled centrally.


10. CACHING AND LIGHTWEIGHT PERSISTENCE
=======================================

Files:
- lib/core/storage/page_cache.dart
- lib/core/storage/token_storage.dart

10.1 PageCache

PageCache is a simple SharedPreferences-backed JSON cache.

Pattern used by many repositories:

1. try live API request
2. save result to cache
3. if request fails, read cached result
4. if cache exists, return cached model
5. otherwise rethrow error

Benefits:
- app can still display previously fetched data during network issues
- repeated entry to common screens can feel faster

Tradeoff:
- there is no visible cache expiration strategy in the shared cache helper
- teams should be aware that stale data can appear if backend changes often


11. NAVIGATION STRATEGY
=======================

Files:
- lib/core/navigation/app_deep_link_navigator.dart
- lib/core/navigation/app_route_observer.dart
- lib/core/presentation/screens/web_page_screen.dart

Navigation style:

- named routes only for a small subset of entry screens

11.1 AppDeepLinkNavigator

This class is one of the most important architectural pieces in the app.

It maps incoming URI paths to native screens such as:

- user profiles
- updates feed
- update detail
- posts feed
- post detail
- blogs feed
- blog detail
- library list and details
- messages list and threads
- groups list and group thread
- birthdays
- search
- inspiration inbox and detail

Fallback behavior:
- if a deep link is valid but not mapped to a native screen, it opens a WebPageScreen with the corresponding web URL

Why this is important:
- the app is not only screen-driven
- it is also URL-driven
- this is critical for notifications, shared links, and cross-platform consistency with the web product

Responsibilities:

- manage external user ID

Important app.dart integration:

13. APP LIFECYCLE BEHAVIOR
==========================

HopefulMeApp observes Flutter lifecycle changes and reacts in several important ways.

13.1 Presence tracking

When the user is authenticated and the app is foregrounded:
- a timer pings the backend every 90 seconds
- this likely supports online presence or "active now" style behavior

13.2 Version check

The app checks:
- after first frame
- when auth state changes into authenticated
- when the app resumes

Version endpoint:
- GET {baseUrl}/api/app/version

Possible outcomes:
- force update dialog
- soft update prompt shown once per session
- no prompt if app is current

13.3 Deep link queueing

The app keeps pending deep links and drains them when:
- the navigator context is ready
- auth state allows the route to be opened

This prevents startup race conditions between app initialization and navigation.

17. DATA FLOW PATTERN
=====================

The project generally follows this flow:

UI Screen/Widget
-> Controller or screen state logic
-> Feature Repository
-> AuthRepository or ApiClient
-> Backend API
-> JSON model parsing
-> UI rebuild

Common implementation pattern:

1. Screen requests data from repository.
2. Repository calls AuthRepository or ApiClient.
3. Response is converted into strongly typed models.
4. Repository may cache the response.
5. Screen renders typed data.

This is a simple and understandable architecture for teams that prefer explicit code over heavy state-management frameworks.


18. STATE MANAGEMENT APPROACH
=============================

The app does not appear to use Provider, Riverpod, Bloc, Cubit, Redux, or GetX as its main state layer.

Instead it uses:
- StatefulWidget state
- ChangeNotifier
- AnimatedBuilder
- constructor dependency injection
- Future-based screen loading
- ValueNotifier for some isolated UI state

Strengths of this approach:
- easy to trace
- fewer abstractions
- straightforward for small to medium teams

Tradeoffs:
- some screens can grow very large
- dependency passing can become verbose
- global coordination becomes harder as the app grows

Current practical advice:
- keep app-wide state in well-defined controllers/services
- keep feature logic in repositories
- avoid putting API logic directly in widgets
- consider introducing a stronger state pattern only if team size, complexity, or testing burden increases

23. HOW TO ADD A NEW FEATURE CORRECTLY
======================================

Recommended structure:

lib/features/new_feature/
|- data/
|  |- new_feature_repository.dart
|- models/
|  |- new_feature_model.dart
|- presentation/
|  |- screens/
|  |- widgets/
|  |- controllers/   (only if needed)

Recommended process:

1. Add API methods inside a repository, not directly inside the screen.
2. Add JSON parsing models in models/.
3. Build screen and widgets in presentation/.
4. If the feature needs app-wide entry access, wire the repository in app.dart.
5. If the feature must open from links or notifications, update AppDeepLinkNavigator and any notification payload handling.
6. If the feature needs shared reusable UI, move only the generic pieces into core/widgets.


24. HOW TO TRACE A BUG IN THIS PROJECT
======================================

If the bug is auth-related:
- start at auth_controller.dart
- then auth_repository.dart
- then api_client.dart

If the bug is home-shell related:
- start at home_screen.dart
- then inspect the repository feeding that section

If the bug is deep-link related:
- start at app.dart deep link setup
- then app_deep_link_navigator.dart

If the bug is notification-related:
- start at onesignal_service.dart
- then app.dart notification tap handler

If the bug is incorrect data rendering:
- inspect feature models first
- then repository parsing
- then widget rendering

27. CURRENT ARCHITECTURAL RISKS OR GROWTH AREAS
===============================================

- app.dart is a heavy composition root and may grow further
- home_screen.dart is very large and acts as a shell plus feature coordinator
- there is minimal automated test coverage
- cache invalidation strategy is simple and may allow stale data
- manual dependency passing may become noisy as more features are added

These are not failures. They are the main areas the team should keep an eye on as the codebase grows.


28. RECOMMENDED TEAM GUIDELINES
===============================

For consistency, the team should try to follow these rules:

- keep API calls inside repositories
- keep model parsing out of UI widgets
- keep feature-specific widgets inside the owning feature
- keep cross-feature reusable widgets in core/widgets
- update deep-link handling whenever a new URL-addressable screen is introduced
- update notification routing whenever backend notification payloads change
- prefer extending existing repositories over duplicating network calls in screens
- keep app.dart focused on composition and global lifecycle logic
- split oversized screens into smaller widgets before they become unmaintainable
